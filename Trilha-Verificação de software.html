Verificação de software

No campo da ciência da computação, verificação de modelos (do inglês, Model Checking) refere-se ao problema de testar automaticamente se um modelo que representa um sistema atende a uma dada especificação.


Fonte: ResearchGate

Tipicamente estes são sistemas de hardware, software e protocolos de comunicação e suas especificações contém requisitos de segurança como a ausência de deadlocks e comportamentos similares que podem levar o sistema a travar. Esse deadlock é uma situação onde duas ou mais ações simultaneamente esperam a finalização da outra para sua continuação, e assim nenhuma delas pode ser concluída.

Para resolver tal problema de maneira algorítmica, tanto o modelo do sistema quanto a especificação são formulados em alguma linguagem matematicamente precisa. Para esta finalidade, ela deve ser formulada como uma tarefa de lógica, de forma a verificar se uma dada estrutura satisfaz uma dada fórmula lógica. O conceito é geral e se aplica a todos os tipos de lógicas e estruturas apropriadas. Um problema simples de verificação de modelo é verificar se uma dada fórmula na lógica proposicional é satisfeita por uma dada estrutura.

Fonte: Stack Overflow

Técnicas aplicadas na verificação formal:

Satisfiability Modulo Theories (SMT): refere-se ao problema de determinar se uma fórmula em lógica de primeira ordem pode ser satisfeita no que diz respeito a alguma teoria lógica;

Bounded Model Checking (BMC): algoritmos para explorar o espaço de estado de um sistema de transição para determinar se ele obedece a certa especificação.

O custo dos erros nos requisitos costuma ser alto, exigindo pelo menos retrabalho e manutenção. Se você implementar os requisitos incorretos como eles estão, isso pode levar a um comportamento incorreto do sistema no campo e altos custos, como perda de vidas e propriedades, especialmente em tempo real, sistemas integrados de segurança crítica. Problemas semelhantes existem para garantir a qualidade do projeto do sistema.

Uma maneira de melhorar a qualidade de seus requisitos e design é usar ferramentas automatizadas para verificar a qualidade de vários aspectos dos requisitos e design. Mas quais ferramentas? Construir ferramentas para verificar requisitos ou projetos escritos em inglês é claramente extremamente difícil. É necessário impor uma linguagem formal clara, rigorosa e inequívoca para declarar os requisitos. Se a linguagem para escrever requisitos e design tiver uma semântica bem definida, pode ser viável desenvolver ferramentas para analisar as declarações escritas nessa linguagem. Essa ideia básica de usar uma linguagem rigorosa para escrever requisitos ou design é agora reconhecida como uma base para a verificação do sistema.

Uma ferramenta de verificação de modelo aceita os requisitos ou projeto do sistema (chamados modelos) e uma propriedade (chamada especificação) que se espera que o sistema final satisfaça. A ferramenta, então, retorna sim se o modelo fornecido satisfaz as especificações fornecidas e gera um contra-exemplo de outra forma. O contra-exemplo detalha por que o modelo não atende à especificação. Ao estudar o contra-exemplo, você pode localizar a origem do erro no modelo, corrigir o modelo e tentar novamente. A ideia é que, ao garantir que o modelo satisfaça as propriedades do sistema suficientes, aumentamos nossa confiança na correção do modelo. Os requisitos do sistema são chamados de modelos porque representam requisitos ou design.

Então, qual linguagem formal funciona para definir modelos? Não há uma resposta única, uma vez que os requisitos (ou design) para sistemas em diferentes domínios de aplicativo variam muito. Por exemplo, os requisitos de um sistema bancário e aeroespacial diferem em tamanho, estrutura, complexidade, natureza dos dados do sistema e operações realizadas. Em contraste, a maioria dos sistemas embarcados em tempo real ou de segurança crítica são orientados para controle em vez de orientados para dados, o que significa que o comportamento dinâmico é muito mais importante do que a lógica de negócios (a estrutura e as operações nos dados internos mantidos pelo sistema). Esses sistemas orientados para controle ocorrem em uma ampla variedade de domínios: aeroespacial, aviônica, automotivo, instrumentação biomédica, automação industrial e controle de processo, ferrovias, usinas nucleares e assim por diante.

Para sistemas orientados ao controle, as máquinas de estado finito (FSM) são amplamente aceitas como uma notação boa, limpa e abstrata para definir requisitos e design. Mas é claro, um FSM “puro” não é adequado para modelar sistemas industriais complexos da vida real.
